## Codebase Patterns
- TinyFusion is a Rust workspace with two crates: tinyfusion-core and tinyfusion-gui
- Core uses Axum for HTTP server, Tokio for async runtime
- GUI uses Tauri v2 with React frontend
- Design system follows refined developer console aesthetic (dark-first, Inter font, Lucide icons)
- `tracing_subscriber::fmt().init()` must be called in main.rs for `info!`/`tracing` logs to appear on stdout
- Use `tower::ServiceExt` with `axum::body::to_bytes` for unit testing routes without starting a real server
- Frontend: Vite + React + TypeScript + Tailwind CSS v4 + React Router v7 + Lucide React
- Tailwind v4: `@import "tailwindcss"` in CSS + `@tailwindcss/vite` plugin in vite.config.ts (no config.js needed)
- Design system CSS variables defined in `frontend/src/index.css` — all components use `var(--token-name)`
- `npx vite build` produces `dist/` which Tauri bundles via `frontendDist` in tauri.conf.json
- Google Fonts @import must be the first rule in index.css

- **Learnings for future iterations:**
  - Handler functions using `axum::Json<T>` need T to be at least `pub(crate)` — private types cause compilation errors when the handler is `pub`

  - The validate() approach (separate function returning Result<String, String>) is cleaner than deriving axum Extractors for early validation — easier to test and produces better error messages

  - When generating responses, use `messages.iter().rev().find(|m| m.role == "user")` to get the last user message content, not `.last()` which picks up assistant messages
